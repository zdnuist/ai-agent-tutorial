## 三、遇到的问题及解决方案

### 3.1 问题一：API 限流导致监控中断

**现象：**
```
[ERROR] 2026-03-17 14:30:00 - GitHub API rate limit exceeded
[ERROR] 2026-03-17 14:30:01 - 监控脚本暂停运行
```

**原因分析：**
- 监控频率过高（每 2 小时一次）
- 单次请求数据量过大（per_page=100）
- 未使用条件请求（If-None-Match）

**解决方案：**

```python
# 1. 降低监控频率
MONITOR_INTERVAL = 4 * 3600  # 4 小时

# 2. 使用条件请求
headers = {
    "Authorization": f"token {TOKEN}",
    "If-None-Match": cached_etag  # 使用缓存的 ETag
}
response = requests.get(url, headers=headers)
if response.status_code == 304:
    return  # 无变化，不处理

# 3. 分页获取，减少单次请求量
for page in range(1, 4):  # 最多 3 页
    params = {"page": page, "per_page": 30}
    ...
```

**效果：** API 调用减少 70%，不再触发限流。

### 3.2 问题二：PR 审查意见理解偏差

**现象：**
- 维护者要求"添加更多测试"
- 我添加了 5 个单元测试，但仍被要求补充
- 来回沟通 3 轮，耗时 2 天

**原因分析：**
- 没有理解维护者的真实需求
- 测试覆盖的是"正常路径"，但维护者关心"边界情况"
- 沟通时没有主动询问具体需求

**解决方案：**

**改进后的沟通话术：**
```
@maintainer 感谢您的审查！关于测试补充，我想确认一下：

1. 您希望重点覆盖哪些边界情况？
2. 是否需要集成测试，还是单元测试即可？
3. 有特定的测试框架偏好吗？

我想确保补充的测试符合项目要求，谢谢！
```

**效果：** 维护者明确回复需要"异常输入测试"，1 小时内完成补充，当天合并。

**经验总结：**
> 当审查意见模糊时，主动询问比盲目修改更高效。

### 3.3 问题三：多任务并行导致注意力分散

**现象：**
- 同时处理 4 个 PR
- 每个都进展缓慢
- 晚上复盘发现有效工作时间不足 50%

**原因分析：**
- 任务切换成本高（平均 15 分钟/次）
- 没有明确的优先级排序
- 害怕错过"好任务"，来者不拒

**解决方案：**

**引入 WIP（Work In Progress）限制：**
```python
# 配置文件中添加
MAX_CONCURRENT_PRS = 3  # 最多同时进行 3 个 PR

# 任务队列管理
class TaskQueue:
    def __init__(self):
        self.active = []  # 进行中
        self.pending = []  # 等待中
        
    def can_accept(self) -> bool:
        return len(self.active) < MAX_CONCURRENT_PRS
    
    def add(self, task):
        if self.can_accept():
            self.active.append(task)
        else:
            self.pending.append(task)
            notify_user("任务已加入等待队列")
```

**时间盒规则：**
```
单任务时间上限 = 预估时间 × 1.5

超时处理：
1. 暂停当前工作
2. 评估是否继续或放弃
3. 如继续，申请延期或降低范围
```

**效果：** 任务完成速度提升 40%，工作压力明显降低。
