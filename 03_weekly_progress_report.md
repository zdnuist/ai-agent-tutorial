## 三、遇到的问题及解决方案

### 问题 1：GitHub API Rate Limit

**现象：** 运行监控脚本 1 小时后收到 403 错误

**原因：** 未认证请求限制为 60 次/小时

**解决方案：**
```python
# 使用个人访问令牌
headers = {
    "Authorization": f"token {GITHUB_TOKEN}",
    "Accept": "application/vnd.github.v3+json"
}
# 认证后限额提升至 5000 次/小时
```

**经验：** 务必在 `.env` 中配置 token，不要硬编码到代码中。

### 问题 2：PR 被拒率过高

**现象：** 前 3 个 PR 有 2 个被要求修改

**原因分析：**
- 未阅读项目 CONTRIBUTING.md
- 代码风格与项目不一致
- 缺少测试用例

**解决方案：**
1. 建立 PR 检查清单（见下文）
2. 在本地运行项目的 lint/test 命令
3. 参考该项目历史 PR 的格式

**PR 检查清单：**
```
□ 阅读 CONTRIBUTING.md
□ 运行 pre-commit hooks
□ 添加/更新测试
□ 更新文档（如有需要）
□ PR 描述关联 Issue
□ 代码审查自查
```

### 问题 3：任务信息不对称

**现象：** 认领任务后才发现需要特定领域知识

**解决方案：**
- 建立任务筛选评分系统：
  ```
  可行性评分 = 技术匹配度 (0-5) + 时间充裕度 (0-5) - 知识缺口 (0-5)
  ```
- 只接评分 > 6 的任务
- 对于不确定的任务，先在 Issue 中询问维护者

### 问题 4：时间管理失控

**现象：** 某天花费 6 小时在一个 $30 的任务上

**解决方案：**
- 设置时间盒（Time Boxing）：每个任务最多投入 4 小时
- 超时后重新评估是否继续
- 建立任务放弃机制（沉没成本不是成本）

## 六、给新手的建议

基于本周踩坑经验，给想尝试 GitHub Bounty 的朋友几点建议：

1. **从文档开始**：门槛低、竞争少、建立信心

3. **不要贪多**：一次专注 1-2 个任务
4. **重视沟通**：及时回复审查意见
